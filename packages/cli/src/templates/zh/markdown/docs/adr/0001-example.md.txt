# 使用领域事件进行跨上下文通信

状态：已接受

我们使用异步领域事件（发布到消息代理）进行限界上下文之间的通信，而非同步 HTTP 调用。此决策优先考虑自治性和弹性，而非即时一致性。每个上下文可以独立演化，一个上下文的临时不可用不会阻塞其他上下文。

## 考虑的选项

- **同步 HTTP/REST 调用** - 被拒绝，因为它创建紧耦合和级联故障。如果订单上下文同步调用库存上下文，而库存上下文宕机，整个订单流程将中断。

- **直接数据库访问** - 被拒绝，因为它违反了限界上下文边界，并使模式变更变得危险。多个上下文写入同一张表会导致数据损坏。

- **GraphQL 联邦** - 被拒绝，因为它仍然需要同步可用性，并且对于我们团队当前的专业水平增加了复杂性。

## 后果

- **最终一致性** - 上下文之间的数据可能暂时不同步（例如，结账期间显示的库存数量可能略有过时）。我们通过刷新机制和清晰的 UI 指示器来缓解这一问题。

- **事件模式演化** - 我们必须谨慎地对事件进行版本控制并保持向后兼容性。引入了运维开销。

- **调试复杂性** - 跟踪跨多个事件的流程比跟踪同步调用栈更困难。我们通过关联 ID 和集中式日志记录来解决这个问题。

- **消息代理依赖** - 系统需要消息代理（RabbitMQ）正常运行。这是为了自治性优势而付出的可接受权衡。
