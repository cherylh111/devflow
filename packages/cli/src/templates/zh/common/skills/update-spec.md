# 更新项目规范

把实现、调试或评审过程中发现的可复用约定写回 `.devflow/spec/`。

适用场景：

- 发现项目已有但未文档化的编码模式
- 修复 bug 后得到可复用规则
- 新增跨层契约、数据格式或工作流约定
- 发现某个反模式需要明确禁止

步骤：

1. 确认这条知识是否稳定、可复用，而不是一次性偏好。
2. 找到最合适的 spec 文件：
   - backend 规则写到 `.devflow/spec/<package>/backend/`
   - frontend 规则写到 `.devflow/spec/<package>/frontend/`
   - 跨层或通用思考规则写到 `.devflow/spec/guides/`
3. 用简洁、可执行的语言记录规则。
4. 尽量引用真实代码路径或已发生的问题作为证据。
5. 如果改动影响未来任务加载，确保相关 index 文件能指向新规则。

不要把临时实现细节、个人偏好或未验证猜想写成规范。
