# 打破重复修复循环

当同类 bug 反复出现，或修复后又以相似形式失败时，做一次根因分析。

分析维度：

1. **现象**：用户看到的问题是什么？
2. **直接原因**：是哪段代码或配置触发？
3. **根因类别**：契约不清、状态重复、跨层格式漂移、测试缺失、工具链假设错误，还是其他原因？
4. **为什么之前修复失败**：修复只处理了症状、漏了镜像路径、没有覆盖 update/init 双路径，还是缺少回归测试？
5. **预防机制**：应增加测试、共享 helper、类型约束、spec 规则，还是迁移脚本？

输出要求：

- 先给可验证的根因，而不是猜测。
- 修复代码后，补上能防止复发的验证。
- 如果学到的规则对未来有价值，写入 `.devflow/spec/` 或使用 `devflow-update-spec`。
