# 逻辑原型参考

当未知点是行为而不是展示时，使用逻辑原型。

## 适合的问题

- 这个状态机是否允许不可能的转换？
- 哪个解析或规范化规则能干净处理真实输入？
- 生命周期规则在边界情况下是否仍然容易理解？
- 走完示例后，哪种算法形态更简单？

## 形态

围绕真实想法构建最小可运行 harness：

1. 把原型放在生产路径之外，或明确标记为一次性代码。
2. 只建模回答问题所需的状态和转换。
3. 包含少量硬编码场景或一个很小的交互循环。
4. 每一步后打印完整相关状态。
5. 保持依赖最少，并使用项目原生工具。

## 命令

尽量使用已有项目 runner：

```bash
pnpm tsx path/to/prototype.ts
python path/to/prototype.py
node path/to/prototype.js
```

命令应该容易重跑，也容易删除。

## 避免事项

- 不要加入生产级持久化，除非正在验证的就是持久化。
- 不要把状态藏在抽象后面。
- 不要给一次性代码添加大范围测试套件。
- 不要在 Phase 2 直接复用原型实现。

## Findings 清单

把答案记录到 `research/prototype-<slug>-findings.md`：

- 问题；
- 已演练场景；
- 状态快照或关键输出；
- 决策；
- 清理状态。
