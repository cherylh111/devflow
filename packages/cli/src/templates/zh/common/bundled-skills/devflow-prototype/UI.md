# UI 原型参考

当未知点是布局、交互或视觉比较时，使用 UI 原型。

## 适合的问题

- 哪种布局让主要工作流最容易扫描？
- 哪种交互模型最能暴露所需状态？
- 哪个方案能处理空、加载和高密度数据状态？
- 哪种视觉层级支持重复的操作型使用？

## 形态

优先使用一个 route 或 surface，并能切换方案：

1. 把原型代码放在生产路径之外，或明确标记为一次性代码。
2. 构建二到四个有实质差异的方案。
3. 用 route segment、search param、tab 或简单本地控件切换方案。
4. 使用真实领域数据，而不是通用填充内容。
5. 渲染关键状态：empty、populated、error、loading，以及相关时的 dense。

## 命令

使用应用已有的开发命令：

```bash
pnpm dev
pnpm --filter <package> dev
```

如果项目没有 app server，让原型可通过单个本地命令运行，并在 findings 中记录。

## 避免事项

- 不要打磨与决策无关的视觉细节。
- 不要接入真实写入，除非数据变更正是问题。
- 不要把方案埋在无法快速比较的独立分支里。
- 不要把原型组件直接提升为生产代码。

## Findings 清单

把答案记录到 `research/prototype-<slug>-findings.md`：

- 问题；
- 已比较方案；
- 截图或简洁观察；
- 决策；
- 清理状态。
